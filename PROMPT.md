# PROMPT — bootstrap нового проєкту за методикою MIN

Скопіюй блок між `--- BEGIN ---` і `--- END ---` у Claude / Gemini / інший CLI-агент. Агент проведе коротке інтерв'ю (≤10 хв) і заповнить шаблони реальним змістом.

Працює у двох режимах:
- **Mode A — новий проєкт** з нуля.
- **Mode B — міграція** уже існуючого складного проєкту на методику MIN (як було зроблено для `Server-work-office-2026`).

Який режим — агент визначить за першим питанням.

## Крок 0 — ознайомлення

Спочатку визнач шлях до template-kit:

- Якщо промпт запускається з кореня kit — `<KIT_PATH>` = поточна папка.
- Якщо ні — запитай користувача: "Де лежить AGENT-PROJECT_CREATOR_MIN? Дай абсолютний шлях."
- Не використовуй hard-coded шляхи з чужої машини. Усі команди нижче виконуються через `<KIT_PATH>`.

Спочатку прочитай 12 золотих правил, потім повний стандарт MIN — вони задають **що** і **в якому порядку** робити. Далі ознайомся з шаблонами, які копіюються у новий проєкт.

1. `<KIT_PATH>/STANDART.md` ← 12 золотих правил (концентрат)
2. `<KIT_PATH>/METHODOLOGY.md` ← повний стандарт, SSOT
3. `<KIT_PATH>/template/README.md`
4. `<KIT_PATH>/template/CLAUDE.md`
5. `<KIT_PATH>/template/TASKS.md`
6. `<KIT_PATH>/template/DEMO.md`
7. `<KIT_PATH>/template/simple_words.md`
8. `<KIT_PATH>/template/SESSION.md`
9. `<KIT_PATH>/template/DECISIONS.md`
10. `<KIT_PATH>/template/RISKS.md`
11. `<KIT_PATH>/template/ACCESS.example.md`
12. `<KIT_PATH>/template/docs/00-README.md`

Зрозумій структуру і призначення кожного файлу. Якщо час обмежений — fast path: `STANDART.md` обов'язково, решта 11 файлів читати at-need перед відповідним кроком. Пріоритет при суперечності: `STANDART.md` > `METHODOLOGY.md` > файли в `template/`.

## Mode A — новий проєкт з нуля

Постав питання **по одному**, чекай відповідь перед наступним. Не запитуй все одразу.

A1. **Назва проєкту?** Англійською, без пробілів.
A2. **Куди створити?** Абсолютний шлях. Якщо папка вже існує — зупинись і повідом, не перезаписуй.
A3. **Що це за проєкт?** 2–3 речення звичайною мовою, для нефахівця.
A4. **Яку реальну проблему вирішує?** 1 речення.
A5. **Done when?** 3–7 конкретних, спостережуваних критеріїв. Якщо відповідь "буде працювати" або "користувачі задоволені" — перепитати з конкретикою. Кожен критерій має відповідати на питання "як я перевірю руками".
A6. **Default mode?** READ-ONLY чи WRITE.
A7. **Мова документації?** Українська чи англійська.
A8. **Чого НЕ робити без approval?** 2–4 дії (напр. видалення, deploy, API з оплатою).
A9. **Перші 1–3 задачі.** Назва + Done when для кожної.
A10. **Чи є вже матеріал** (нотатки, файли, посилання), з якого підтягнути контекст? Якщо так — шляхи.

### Дії після інтерв'ю

1. Скопіювати шаблон: `cp -r <KIT_PATH>/template/. <шлях з A2>/`. Зверни увагу на крапку після `template/` — копіюємо **вміст**, а не саму папку `template`. До нового проєкту не потрапляють `PROMPT.md`, `.claude/`, `.DS_Store` і meta-`CLAUDE.md` кореня — це by design.
2. Заповнити **тільки** ці файли у такому порядку:
   - `README.md` — назва (A1), TL;DR (A3+A4), Done when (A5).
   - `CLAUDE.md` — секція "Що це за проєкт" (A3+A4), default mode (A6), заборонені дії (A8), мова (A7).
   - `simple_words.md` — переклад A3+A4 звичайною мовою.
   - `DEMO.md` — приймальний сценарій, що випливає з Done when (A5). Кожен критерій → один крок DEMO.
   - `TASKS.md` — задачі з A9 як P0-001…P0-003, статус READY, заповнені поля Safety, Steps, Done when.
3. **Не заповнювати** (мають лишитись з placeholder-ами):
   - `RISKS.md` — порожній до першої знайденої загрози.
   - `DECISIONS.md` — порожній до першого нетривіального вибору.
   - `SESSION.md` — порожній до першої реальної робочої сесії.
   - `ACCESS.example.md` — лишається як шаблон. Реальний `ACCESS.md` користувач створить сам, коли з'являться секрети.
   - `docs/00-README.md` — лишається як шаблон (правила оформлення `docs/`).
   - `docs/01-domain-template.md` — лишається як шаблон.
4. У кінці видати користувачу summary:
   - Які файли заповнено.
   - Які лишилися як шаблон і **за якої умови** заповнювати кожен.
   - Який наступний крок (зазвичай: "відкрий новий проєкт у Claude і починай з TASKS.md → P0-001").

## Mode B — приведення в порядок існуючого проєкту (міграція + чищення)

Mode B = міграція існуючого складного/брудного проєкту до стандарту MIN. Три проходи з approval-gate між ними. Користувач може зупинитись після будь-якого. Жорсткий контракт: **жодних видалень / перейменувань / редагувань існуючих файлів** без явного approval.

Постав питання по одному.

B1. **Шлях до існуючого проєкту?** Абсолютний.
B2. **Що в ньому погано зрозуміло?** 1–3 речення. Це підкаже, на що звернути увагу при аудиті.
B3. **Чи є critical files, які НЕ можна перейменовувати / переміщати** (зовнішні посилання, скрипти, документація)? Список або "ні". Ці файли захищені від усіх трьох проходів.

### Прохід 1 — аудит (READ-ONLY, нічого не змінюємо)

1. `ls -la`, `find . -type f \( -name "*.md" -o -name "*.txt" \)`, `wc -l *.md`. Прочитати все, що схоже на documentation root (`README*`, `CLAUDE*`, `AGENTS*`, `TASKS*`, `NEXT_ACTIONS*`).
2. Скласти **три таблиці**:
   - **MIN-роль → один кандидат** (`README`, `CLAUDE`, `simple_words`, `TASKS`, `SESSION`, `DECISIONS`, `RISKS`, `DEMO/ACCEPTANCE`, `ACCESS.example`, `docs/`) — `✅ є` / `✅ частково` / `❌ нема` + назва існуючого файлу.
   - **MIN-роль → кілька кандидатів** — дублі (`RISKS.md` + `05-risk-register.md` + `risks_old.md`), треба обрати канонічний.
   - **Не вписується в MIN-роль** — кандидати на сміття: `Untitled*.md`, `Copy of*`, `*~`, `*.bak`, порожні файли, `.DS_Store`, `Thumbs.db`.
3. Показати три таблиці користувачу.
4. Додати секцію **Відкриті питання**, якщо є неоднозначності: канонічний `TASKS.md`, legacy `NEXT_ACTIONS.md`, non-markdown source of truth (`.docx`, `.pages`, `.pdf`), додаткові protected files. Кожне питання має мати конкретні опції для оператора.
5. Запитати: чи вірно зіставлено? Які файли додати у B3 (захищені)? Як відповісти на відкриті питання?
6. **СТОП.** Не починати Прохід 2 без явного "так, переходимо".

### Прохід 2 — `START_HERE.md` + гепи (тільки нові файли)

5. Створити `START_HERE.md` — точка входу з картою `MIN-роль → реальний файл`. Для дублів явно вказати: `канон: X; дублі: Y, Z — Прохід 3`. Якщо є кілька MIN-площин — додати секцію "Площини".
6. Створити відсутні MIN-файли:
   - `ACCEPTANCE.md` (або `DEMO.md`) — критерії завершення. Для не-software проєктів — операційний чекліст.
   - `TASKS.md`, якщо немає файлу зі структурою MIN-TASKS (`Status` / `Safety` / `Done when`). Існуючий operational-формат (`TODO`, `NEXT_ACTIONS`, канбан) залишити як архів історії з приміткою у `START_HERE.md`.
   - `DECISIONS.md`, якщо відсутній — ретроспективно витягнути 3–7 рішень із `PROMPT.md`, `CLAUDE.md`, `README`. Не вигадувати.
   - `SESSION_LOG.md`, якщо відсутній — append-only журнал, перший запис: дата + "MIN migration pass 2 done".
   - `ACCESS.example.md`, якщо реальний секретний файл є, а шаблону без значень нема.
7. `simple_words.md` у Mode B не створюється без окремого інтерв'ю A3/A4 — інакше це вигадування, що порушує правило `UNKNOWN`.
8. Для legacy non-markdown source of truth (`.docx`, `.pages`, `.pdf`) діяти тільки за відповіддю з "Відкритих питань": створити новий `.md` через конверсію (`pandoc` / `textutil`) або лишити як protected. Вихідний файл не редагувати; переміщення в `_attic/` можливе тільки в Проході 3.
9. **Не чіпати** на цьому проході нічого існуючого: ні редагувати, ні перейменовувати, ні переміщати. Тільки створення нових файлів.
10. Запитати approval на Прохід 3 (чищення). Користувач може зупинитись тут — проєкт уже відповідає мінімуму MIN.

### Прохід 3 — карантин (тільки під явний approval, по списку)

11. **Не видаляти, не дедуплікувати контент.** Замість видалення — переміщення через `mv` в `_attic/YYYY-MM-DD/` з збереженням підструктури шляхів. Операція має бути підтверджена оператором і задокументована у `DECISIONS.md`, якщо змінює структуру проєкту.
12. Показати **одним питанням** список кандидатів на карантин: "Перенести в `_attic/<дата>/` ці N файлів? Y/N + які виключити". Не питати файл-за-файлом.
13. Кандидати:
    - Сміття з Проходу 1, таблиця 3 (`Untitled*`, `*~`, `*.bak`, порожні).
    - Дублі MIN-ролей з таблиці 2 (зайві після вибору канона).
14. Якщо оператор явно вимагає `rm -rf` замість карантину — зупинись і постав STOP-gate з 4 опціями: карантин, карантин + архів, irreversible delete, залишити як є. Для irreversible delete потрібна явна approval-фраза; записати у `DECISIONS.md` підставу, список втрачених артефактів (кількість файлів, total size) і факт необоротності.
15. OS-сміття (`.DS_Store`, `Thumbs.db`, `*~`, `*.bak`) можна включити у список карантину або protected-список за рішенням оператора.
16. Виконати пакетом через `mv`, тільки після approval на список.

### Що ніколи не чіпати (жоден прохід)

- `TASKS.md` — навіть якщо великий. Розмір ≠ погана структура.
- Numbered domain files (`01-*.md`, `02-*.md`, `03-*.md`).
- Будь-який файл зі списку B3.
- Реальний `ACCESS.md` / `.env` / `*.key` / `*.pem` — навіть як кандидат на карантин.
- Будь-який не-markdown файл за замовчуванням: код (`*.py`, `*.ts`, `*.go`, `setup.sh`, `requirements.txt`), бінарники, скріншоти, артефакти.
- Папки з даними або артефактами: `input/`, `output*/`, `reports/`, `backup*/`, `photo*/`, bind-mounted директорії.
- Контент існуючих файлів — лише читаємо, не редагуємо.

### Summary після кожного проходу

- Прохід 1: три таблиці + список відкритих питань для уточнення.
- Прохід 2: список створених файлів, підтвердження "жоден існуючий не змінено".
- Прохід 3: список переміщених у `_attic/<дата>/` файлів, нагадування: після `DEMO.md` sign-off — або cleanup `_attic/` з явним approval, або залишити карантин як архів.

## Правила для обох режимів

- **Не вигадувати значення.** Якщо користувач не дав відповіді — `UNKNOWN`, не "ймовірно".
- **Не виконувати дії поза цільовою папкою.** Створювати/копіювати/редагувати лише у межах нового або мігрованого проєкту.
- **Не запускати risky команди** (`rm -rf`, deploy) — це не входить у bootstrap.
- **Зупинись і запитай**, якщо припущення незворотне (перезапис існуючої папки, видалення файлу, масове перейменування).
- **Мова відповіді користувачу** — та сама, що в проєкті (українська за замовчуванням, якщо не сказано інше).
- **У кінці завжди** — короткий summary "що зроблено, що далі".
