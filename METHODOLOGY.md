# METHODOLOGY.md — стандарт MIN

Single source of truth для методики MIN: який мінімальний набір документів має бути в проєкті, у якому порядку з ним працювати, і що робити на початку / в процесі / після завершення.

Якщо текст у [PROMPT.md](PROMPT.md), [template/README.md](template/README.md) чи [template/CLAUDE.md](template/CLAUDE.md) суперечить цьому документу — пріоритет має цей файл, інші треба синхронізувати.

---

## 1. Що таке MIN

Мінімальний набір документів і ритуал, який дозволяє запустити новий проєкт за ≤10 хв і вести його разом із CLI-агентом (Claude Code / Gemini / Cursor) без накопичення розсинхронізованої документації. Один проєкт = одна папка = один стандарт. Жодного monorepo, жодних спільних tooling-шарів між проєктами.

Проєкт може мати 2 паралельні MIN-площини, якщо це справді різні робочі контексти всередині однієї папки: наприклад, інфраструктура + код, backend + frontend, hardware + automation scripts. Кожна площина має мати повний свій MIN-набір або явну карту відповідників. `START_HERE.md` у Mode B мусить містити секцію "Площини", щоб читач бачив межі між ними.

Методика працює у двох режимах:

- **Mode A** — bootstrap нового проєкту з нуля (`cp -r template/.` → інтерв'ю → заповнені файли).
- **Mode B** — міграція уже існуючого складного проєкту (тільки гепи, без переписувань).

Поточний режим зберігається **тільки** в `SESSION.md` → `Current Execution Mode`.

## 2. Обов'язкові документи проєкту

| Файл | Роль | Коли заповнюється | Хто оновлює |
|---|---|---|---|
| `README.md` | TL;DR + Done when + карта документів | bootstrap (раз) | людина |
| `CLAUDE.md` | Правила для агента (mode, дозволено / заборонено) | bootstrap (раз) | людина |
| `simple_words.md` | Словник домену + пояснення без жаргону | bootstrap, оновлюється | людина |
| `DEMO.md` | Приймальний сценарій ("як виглядає працює") | bootstrap (раз) | людина |
| `TASKS.md` | Backlog P0/P1/P2, поля Safety / Steps / Done when | постійно | людина + агент |
| `SESSION.md` | Поточний стан + mode + останні дії (append-only) | кожна сесія | агент після задачі |
| `DECISIONS.md` | ADR-и (нетривіальні вибори) | за фактом | людина + агент |
| `RISKS.md` | Реєстр ризиків + триггери | за фактом | людина + агент |
| `ACCESS.md` (опц.) | Реальні секрети, локально, не для публікації | за фактом | тільки людина |
| `ACCESS.example.md` | Шаблон без значень | bootstrap (як шаблон) | людина |
| `docs/00-README.md` | Карта глибокої документації | за потреби | людина |
| `docs/0N-*.md` | Доменні гайди | за потреби | людина |

**Базове правило:** один факт — один власник. Не дублювати `Current Execution Mode` між `SESSION.md` і `TASKS.md`. Не дублювати критерії приймання між `README.md` і `DEMO.md` (у README — список Done when, у DEMO — послідовні кроки, які їх перевіряють).

## 3. Життєвий цикл проєкту

### 3.1 Початок (bootstrap, ~10 хв)

1. Визначити `<KIT_PATH>` — абсолютний шлях до локального template-kit `AGENT-PROJECT_CREATOR_MIN`.
2. Скопіювати шаблон: `cp -r <KIT_PATH>/template/. <шлях нового проєкту>/` (крапка обов'язкова — копіюємо вміст).
3. Прогнати [PROMPT.md](PROMPT.md) Mode A в CLI-агенті: відповісти на 10 коротких питань (A1–A10).
4. Агент заповнює: `README.md`, `CLAUDE.md`, `simple_words.md`, `DEMO.md`, `TASKS.md`.
5. Агент **не чіпає**: `RISKS.md`, `DECISIONS.md`, `SESSION.md`, `ACCESS.example.md`, `docs/00-README.md`, `docs/01-domain-template.md` — вони лишаються як шаблон.
6. Перший запис у `SESSION.md`: `Current Execution Mode: READ-ONLY`, дата, "bootstrap completed".

**Done when для bootstrap:** усі п'ять заповнених файлів пройшли перечитування ("читач без контексту розуміє за 5 хв, що це і як перевірити, що працює").

### 3.2 Процес (кожна робоча сесія)

Кожна сесія = атомарний цикл:

1. **Відкрити `SESSION.md`** — перечитати поточний `Current Execution Mode` і останній запис ("де зупинились").
2. **Відкрити `TASKS.md`** — взяти 1 пункт з найвищим пріоритетом серед `Status: READY` + `Safety: READ-ONLY` (або WRITE, якщо mode дозволяє).
3. **Перевірити Done when задачі** — якщо порожнє, спершу заповнити критерій, тільки потім починати.
4. **Виконати** — змінити статус задачі на `IN-PROGRESS`.
5. **Зафіксувати рішення** — якщо ухвалено необоротний вибір (бібліотека, формат даних, архітектура), новий запис `D-NNN` у `DECISIONS.md`.
6. **Зафіксувати ризики** — якщо виявлено загрозу (security, vendor lock-in, data loss), новий запис `R-NNN` у `RISKS.md`.
7. **Закрити задачу** — у `TASKS.md`: `[ ]` → `[x]`, додати запис у `Done` ledger.
8. **Закрити сесію** — у `SESSION.md` append: що зроблено, які артефакти, next step (зазвичай ID наступної задачі).

**Правило одного кроку:** не починати другу задачу, поки попередня не закрита через п.7. Якщо застряг → статус `BLOCKED`, причина в полі задачі, перехід до наступної READY.

### 3.3 Завершення (sign-off проєкту або фази)

1. Пройти `DEMO.md` від першого до останнього кроку без помилок. Кожен крок дає спостережуваний результат.
2. Перевірити `README.md` → секцію `Done when`: усі чекбокси відмічені.
3. Закрити всі `P0` у `TASKS.md` — або явно перенести в секцію `Won't do` з причиною.
4. Фінальний запис у `SESSION.md`: статус (`DONE` / `FROZEN`), посилання на демо / деплой / прийом-передачу.
5. Якщо проєкт переходить у "заморожений" стан — додати у локальний `<PROJECTS_ROOT>/SYSTEM/FOCUS_PLAN.md` секцію FROZEN з датою.
6. Якщо проєкт переходить у "архів" — перенести папку в локальний `<PROJECTS_ROOT>/ARCHIVE/` і додати рядок у `<PROJECTS_ROOT>/SYSTEM/PROJECT_INDEX.md`.

### 3.4 Приведення в порядок legacy-проєкту (Mode B = міграція + чищення)

Застосовується, коли треба привести **уже існуючий складний/брудний** проєкт до стандарту MIN: дублі, `Untitled.md`, забуті чернетки, файли без явної ролі. Контракт жорсткий: жодних видалень / перейменувань без явного approval на кожен крок. Три проходи з approval-gate між ними, користувач може зупинитись після будь-якого.

**Прохід 1 — аудит (READ-ONLY, нічого не змінюємо).**

1. `ls -la`, `find . -type f \( -name "*.md" -o -name "*.txt" \)`, `wc -l *.md` — реальна карта файлів.
2. Скласти **три таблиці** (показати користувачу):
   - **MIN-роль → один кандидат** — все OK, дублів нема.
   - **MIN-роль → кілька кандидатів** — дублі, треба вибрати канонічний.
   - **Не вписується в MIN-роль** — кандидати на сміття (`Untitled*.md`, `Copy of*`, `*~`, `*.bak`, порожні файли, `.DS_Store`, `Thumbs.db`).
3. Додати секцію **Відкриті питання**, якщо є неоднозначність: кілька канонів для однієї MIN-ролі, legacy `NEXT_ACTIONS.md`, non-markdown source of truth, protected files.
4. Запитати: чи вірно зіставлено? Які файли захищені від будь-яких дій (B3 у [PROMPT.md](PROMPT.md))? Як відповісти на відкриті питання?
5. Не переходити до Проходу 2, поки користувач не підтвердив таблиці.

**Прохід 2 — `START_HERE.md` + гепи (тільки нові файли, нічого не чіпаємо).**

6. Створити `START_HERE.md` — точку входу з картою `MIN-роль → реальний файл`. Для дублів явно вказати: `канон: X; дублі: Y, Z — Прохід 3`. Якщо є кілька MIN-площин — додати секцію "Площини".
7. Створити відсутні файли MIN-ролей (`ACCEPTANCE.md`, `DECISIONS.md`, `SESSION_LOG.md`, `ACCESS.example.md`).
8. Якщо немає файлу зі структурою MIN-TASKS (`Status` / `Safety` / `Done when`) — створити `TASKS.md`. Legacy operational-формат (`TODO`, `NEXT_ACTIONS`, канбан) лишити як історичний документ і позначити у `START_HERE.md`.
9. `simple_words.md` у Mode B не створюється без окремого інтерв'ю A3/A4 — інакше це вигадування.
10. Перший запис у `SESSION_LOG.md`: дата, "MIN migration pass 2 done".
11. Запитати approval на Прохід 3 (чищення). Користувач може зупинитись тут — проєкт уже відповідає мінімуму MIN, просто з legacy-шумом всередині.

**Прохід 3 — карантин (тільки під явний approval, по списку).**

12. **Не видаляти, не дедуплікувати контент.** Замість видалення — переміщення через `mv` в `_attic/YYYY-MM-DD/` з збереженням підструктури шляхів.
13. Показати **одним питанням** список кандидатів: "Перенести в `_attic/2026-05-23/` ці N файлів? Y/N + які виключити". Не питати файл-за-файлом.
14. Якщо оператор явно вимагає `rm -rf` замість карантину — STOP-gate з 4 опціями: карантин, карантин + архів, irreversible delete, залишити як є. Для irreversible delete записати у `DECISIONS.md` підставу, кількість файлів і total size.
15. OS-сміття (`.DS_Store`, `Thumbs.db`, `*~`, `*.bak`) можна включити у список карантину або protected-список за рішенням оператора.
16. Дублі MIN-ролей (зайві після Проходу 2): канонічний лишається, інші — в `_attic/YYYY-MM-DD/`.
17. **Не зливати контент дублів автоматично.** Два файли з 90% однаковим текстом можуть мати 10% критичних відмінностей. Merge — задача людини з контекстом, не CLI-агента.

**Обробка non-markdown джерел.**

- `.docx`: створити новий `.md` через `pandoc -f docx -t gfm <source.docx> -o <target.md>`, якщо `pandoc` доступний. Якщо ні — на macOS fallback `textutil -convert txt` і ручне доведення markdown.
- `.pages`: спершу експортувати в `.docx` або `.pdf` руками / через доступний локальний інструмент, потім конвертувати в `.md`.
- `.pdf`: витягнути текст у `.md` тільки якщо PDF є джерелом правди і оператор підтвердив конверсію; скани без OCR позначити `NOT VERIFIED`.
- Вихідний binary-файл зберігається як backup. Його можна перемістити в `_attic/` лише в Проході 3 після approval.
- Після `pandoc` очікуваний manual fixup: HTML-tables перетворити у GFM-tables, якщо таблиці є суттєвим змістом.

**Що з `_attic/` далі.**

- Лишається у working tree до проходження `DEMO.md` / `ACCEPTANCE.md`.
- Після sign-off — або cleanup `_attic/` з approval, або залишити quarantine як архівний шар.
- Не комітити порожнє `_attic/`. Не лишати його у проєкті назавжди — borging-shadow захаращує view.

**Чого не робити в Mode B (жодного проходу).**

- Не використовувати `rm` напряму без irreversible STOP-gate. За замовчуванням — `mv` у `_attic/`.
- Не чіпати `TASKS.md` (навіть якщо великий — розмір ≠ погана структура).
- Не чіпати numbered domain files (`01-*.md`, `02-*.md`).
- Не чіпати файли з B3 (critical files від користувача — зовнішні посилання, скрипти).
- Не чіпати non-markdown файли і data/artifact папки за замовчуванням: код, бінарники, скріншоти, `input/`, `output*/`, `reports/`, `backup*/`, `photo*/`.
- Не редагувати контент існуючих файлів — лише створювати нові або переміщати в карантин.
- Не починати наступний прохід без явного approval на попередній.

## 4. Два режими виконання (Mode A / Mode B)

| | Mode A | Mode B |
|---|---|---|
| Контекст | Нова порожня папка | Існуючий складний проєкт |
| Дія агента | `cp -r <KIT_PATH>/template/.` + заповнення | Аудит + створення тільки гепів |
| Існуючі файли | Немає | Не чіпати, не перейменовувати |
| Виходить | Заповнений `README/CLAUDE/simple_words/DEMO/TASKS` | `START_HERE.md` + карта `MIN-роль → реальний файл` + нові файли в гепах |
| Тривалість | ~10 хв | ~20–30 хв (залежить від розміру проєкту) |

Поточний режим живе в `SESSION.md` → `Current Execution Mode`. Він **не** дублюється в `TASKS.md` (single source of truth).

## 5. Чого не робити

- **Не плодити нові `.md`** без потреби. Методика MINімальна. Якщо інформація не вписалася у 12 файлів з розділу 2 — це сигнал перечитати існуючі, не створювати тринадцятий.
- **Не дублювати інформацію** між файлами. Якщо назва проєкту змінилась — оновити у `README.md`. Якщо в `simple_words.md` стояла стара назва — оновити теж, але не дублювати означення.
- **Не "виправляти" placeholder-и** у `template/` прикладами (`<назва проєкту>` → `Anna-News` — НІ). Шаблон має лишатися шаблоном.
- **Не публікувати секрети** — `ACCESS.md`, `.env`, `*.key`, `*.pem`. Реальні значення лишаються локальними.
- **Не змінювати існуючі файли в Mode B** — навіть якщо здається, що "так буде краще". Mode B створює тільки гепи.
- **Не покладатись на auto-memory агента** замість файлів. Локальні файли проєкту мають пріоритет над будь-якою глобальною пам'яттю. Memory суперечить файлу → оновлюємо файл.
- **Не виконувати задачі зі `Safety: RISKY`** без явного одноразового approval. "На весь день" — не approval.

## 6. Перевірка коректності стандарту (dogfooding)

У цього template-kit немає `build`/`lint`/`test`. Перевірка — це прогін промпту на чистій папці:

1. `mkdir -p /tmp/min-test-A && cd /tmp/min-test-A`
2. Запустити блок з [PROMPT.md](PROMPT.md) в окремій Claude-сесії, обрати **Mode A**.
3. Пройти інтерв'ю A1–A10 до кінця.
4. Переконатись, що отримана папка містить:
   - заповнені (без `<placeholder>`): `README.md`, `CLAUDE.md`, `simple_words.md`, `DEMO.md`, `TASKS.md`;
   - незмінені (як шаблон): `RISKS.md`, `DECISIONS.md`, `SESSION.md`, `ACCESS.example.md`, `docs/00-README.md`, `docs/01-domain-template.md`;
   - без примусової ініціалізації зовнішніх інструментів версіонування.
5. Окремо повторити для **Mode B** на копії реального проєкту (напр. `cp -r <шлях існуючого проєкту> /tmp/min-test-B`).
6. Якщо щось пішло не так — це баг або в [PROMPT.md](PROMPT.md), або в [template/](template/), не в новому проєкті.

## 7. Зв'язки між документами стандарту

- `METHODOLOGY.md` (цей файл) — стандарт, **що** і **в якому порядку**.
- [PROMPT.md](PROMPT.md) — оперативна інструкція для CLI-агента, **як** виконати bootstrap / міграцію.
- [template/README.md](template/README.md) — карта документів для користувача нового проєкту.
- [template/CLAUDE.md](template/CLAUDE.md) — операційні правила всередині нового проєкту (mode, дозволено / заборонено).
- [CLAUDE.md](CLAUDE.md) (root) — meta-правила роботи з самим template-kit.

При додаванні нового шаблонного файлу в [template/](template/) оновити в такому порядку: розділ 2 цього документа → [PROMPT.md](PROMPT.md) Крок 0 (список читання) → [PROMPT.md](PROMPT.md) Mode A крок 2/3 (заповнюється / лишається як шаблон) → [template/README.md](template/README.md) (таблиця документів).
